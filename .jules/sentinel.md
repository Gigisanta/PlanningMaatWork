## 2024-11-20 - [XSS Prevention in User-Editable HTML Preview]
**Vulnerability:** The application renders user-editable HTML directly into an `iframe` without any sandboxing, creating a potential XSS vulnerability if a user adds malicious scripts to the `editableHTML` state.
**Learning:** The `PortfolioPreview` component allows users to edit the generated HTML code. If this HTML contains a `<script>` tag, it would be executed within the context of the application since the `iframe` had no restrictions.
**Prevention:** Adding a `sandbox` attribute to the `iframe` (`sandbox="allow-same-origin allow-popups"`) restricts the capabilities of the content inside the iframe. Importantly, by omitting `allow-scripts`, it prevents any embedded JavaScript from executing, thus mitigating the XSS risk while still allowing the preview to render correctly and allowing necessary features like popups and same-origin access for PDF generation.
## 2024-11-20 - [XSS via Unsanitized Template Literals in HTML Generation]
**Vulnerability:** The `generatePlanHTML` function in `src/lib/generatePlan.ts` builds HTML strings using direct interpolation of user inputs (e.g., `${data.profesion}`). This is a classic Cross-Site Scripting (XSS) vulnerability. If a user sets their profession to `<script>alert(1)</script>`, this script is directly embedded into the generated PDF/HTML view.
**Learning:** Even if data is intended for PDF generation or an internal dashboard, if it passes through an HTML construction phase without a dedicated templating engine (like React/JSX which auto-escapes), manual sanitization is required.
**Prevention:** Implement and use an `escapeHtml` utility function for all text interpolation, and a `safeUrl` function for all `href` and `src` attributes. For URLs injected into attributes, wrap the output of `safeUrl` in `escapeHtml` (e.g. `escapeHtml(safeUrl(url))`) to prevent attribute breakout attacks. Ensure these functions gracefully handle non-string and nullish inputs to avoid crashing the server.
