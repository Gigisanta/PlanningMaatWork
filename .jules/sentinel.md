## 2024-11-20 - [XSS Prevention in User-Editable HTML Preview]
**Vulnerability:** The application renders user-editable HTML directly into an `iframe` without any sandboxing, creating a potential XSS vulnerability if a user adds malicious scripts to the `editableHTML` state.
**Learning:** The `PortfolioPreview` component allows users to edit the generated HTML code. If this HTML contains a `<script>` tag, it would be executed within the context of the application since the `iframe` had no restrictions.
**Prevention:** Adding a `sandbox` attribute to the `iframe` (`sandbox="allow-same-origin allow-popups"`) restricts the capabilities of the content inside the iframe. Importantly, by omitting `allow-scripts`, it prevents any embedded JavaScript from executing, thus mitigating the XSS risk while still allowing the preview to render correctly and allowing necessary features like popups and same-origin access for PDF generation.
## 2024-11-20 - [Insecure Random ID Generation in API Routes]
**Vulnerability:** Several API routes (`outbound/route.ts`, `outbound/[action]/route.ts`, `leads/route.ts`, `classify/route.ts`) were using a predictable combination of `Date.now()` and `Math.random()` to generate IDs for leads and outreach logs (`Date.now().toString(36) + Math.random().toString(36).substr(2, 9)`). This makes the IDs vulnerable to prediction and brute-forcing, posing an Insecure Direct Object Reference (IDOR) risk if these IDs are ever used for access control.
**Learning:** `Math.random()` is not cryptographically secure and should never be used for security-sensitive operations like generating unique identifiers. It is predictable and can be reverse-engineered.
**Prevention:** Replaced the insecure implementation with Node.js's native `crypto.randomUUID()`, which generates cryptographically secure, unpredictable UUIDv4 strings. This is a robust and standard way to generate IDs across the backend.
