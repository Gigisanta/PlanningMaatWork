## 2024-11-20 - [XSS Prevention in User-Editable HTML Preview]
**Vulnerability:** The application renders user-editable HTML directly into an `iframe` without any sandboxing, creating a potential XSS vulnerability if a user adds malicious scripts to the `editableHTML` state.
**Learning:** The `PortfolioPreview` component allows users to edit the generated HTML code. If this HTML contains a `<script>` tag, it would be executed within the context of the application since the `iframe` had no restrictions.
**Prevention:** Adding a `sandbox` attribute to the `iframe` (`sandbox="allow-same-origin allow-popups"`) restricts the capabilities of the content inside the iframe. Importantly, by omitting `allow-scripts`, it prevents any embedded JavaScript from executing, thus mitigating the XSS risk while still allowing the preview to render correctly and allowing necessary features like popups and same-origin access for PDF generation.## 2024-03-21 - [XSS via Unescaped URL Interpolation in HTML Template]
**Vulnerability:** Directly interpolating user-controlled URLs (like `link.url`) into the `href` attribute of anchor tags in an HTML template allowed attackers to inject `javascript:` URIs, leading to a Cross-Site Scripting (XSS) vulnerability.
**Learning:** Pure functions generating HTML via template literals do not have automatic escaping like JSX/React. Thus, attributes such as `href` are susceptible to both protocol-based XSS (e.g., `javascript:`) and attribute-breakout XSS if quotes are not escaped.
**Prevention:** Always wrap interpolated URLs with an explicit URL sanitizer (`safeUrl` to validate protocols) and an HTML escaper (`escapeHtml` to encode quotes and entities), resulting in a pattern like `href="${escapeHtml(safeUrl(url))}"` when building raw HTML templates.
