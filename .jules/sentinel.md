## 2024-11-20 - [XSS Prevention in User-Editable HTML Preview]
**Vulnerability:** The application renders user-editable HTML directly into an `iframe` without any sandboxing, creating a potential XSS vulnerability if a user adds malicious scripts to the `editableHTML` state.
**Learning:** The `PortfolioPreview` component allows users to edit the generated HTML code. If this HTML contains a `<script>` tag, it would be executed within the context of the application since the `iframe` had no restrictions.
**Prevention:** Adding a `sandbox` attribute to the `iframe` (`sandbox="allow-same-origin allow-popups"`) restricts the capabilities of the content inside the iframe. Importantly, by omitting `allow-scripts`, it prevents any embedded JavaScript from executing, thus mitigating the XSS risk while still allowing the preview to render correctly and allowing necessary features like popups and same-origin access for PDF generation.## 2026-04-23 - [Insecure Random ID Generation]
**Vulnerability:** The API routes generated unique identifiers using a highly predictable pattern: `Date.now().toString(36) + Math.random().toString(36).substr(2, 9)`.
**Learning:** While this approach creates visually unique strings, `Math.random()` is not cryptographically secure, and the timestamp portion makes the IDs highly predictable. If these IDs are used for sensitive operations (like accessing specific lead data or outreach logs), an attacker could predict valid IDs and execute Insecure Direct Object Reference (IDOR) attacks.
**Prevention:** Always use cryptographically secure methods for generating unique identifiers. In Node.js environments (like Next.js API routes), use the native `crypto.randomUUID()` which generates secure UUIDv4 strings without requiring external dependencies like `uuid`.
