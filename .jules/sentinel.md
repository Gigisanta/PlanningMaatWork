## 2024-11-20 - [XSS Prevention in User-Editable HTML Preview]
**Vulnerability:** The application renders user-editable HTML directly into an `iframe` without any sandboxing, creating a potential XSS vulnerability if a user adds malicious scripts to the `editableHTML` state.
**Learning:** The `PortfolioPreview` component allows users to edit the generated HTML code. If this HTML contains a `<script>` tag, it would be executed within the context of the application since the `iframe` had no restrictions.
**Prevention:** Adding a `sandbox` attribute to the `iframe` (`sandbox="allow-same-origin allow-popups"`) restricts the capabilities of the content inside the iframe. Importantly, by omitting `allow-scripts`, it prevents any embedded JavaScript from executing, thus mitigating the XSS risk while still allowing the preview to render correctly and allowing necessary features like popups and same-origin access for PDF generation.
## 2024-11-20 - [String Interpolation XSS in HTML Generation]
**Vulnerability:** The `generatePlanHTML` function in `src/lib/generatePlan.ts` manually constructs an HTML document string using string interpolation. User inputs (like `profesion`, `objetivo`, `asesorNombre`) and URLs were injected directly without sanitization, leading to a critical XSS vulnerability.
**Learning:** Even when outputting to an iframe or exporting as a PDF, manual HTML construction creates XSS risks if users can control any of the interpolated data. In particular, injecting data into `href` attributes requires careful protocol validation to prevent `javascript:` execution.
**Prevention:** Introduced an `escapeHtml` utility to encode HTML special characters for all text-based interpolations, and a `safeUrl` utility to strip dangerous protocols before injecting links into `href` attributes. Crucially, when rendering trusted HTML fragments (like `platformLinksHTML`), the fragment itself must not be escaped, but the individual variables that construct it must be.
